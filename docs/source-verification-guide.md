# ドキュメント・仕様作成時の再発防止ガイド

このドキュメントは、プロジェクトで発生した「架空の情報ソース記載」を防止するためのチェックリストと手順です。

## 背景

プロジェクト v2.5.0 で以下の問題が発生しました：

- **問題**: `docs/test-specifications-sources.md` に、実装ファイルに存在しない「ユーザー質問への対応パターン」の仕様を、架空の情報ソースと共に記載
- **原因**: 実装ファイルを確認せず、AGENTS.md の内容を根拠として推測で記載
- **結果**: ドキュメントの信頼性が損なわれ、3 つのファイル（bedrock-agent.yaml、cfn-dev-parameters.json、AGENTS.md）の修正と ドキュメント削除が必要に

## 再発防止方法

### 方法 1：source-verification スキルの使用（推奨）

ドキュメント、仕様書、テスト計画を作成・編集する際は、以下のコマンドでスキルを読み込んでください：

```bash
# OpenCode CLI でスキルを読み込む
ctrl+p  # コマンドパレット
# → "Load Skill: source-verification" を選択
```

**スキルが提供する内容:**
- ドキュメント内のすべての主張に対する情報ソース確認チェックリスト
- 架空の情報源を作成しないための具体的ガイドライン
- よくある誤りと修正例
- 段階的改善の手順

### 方法 2：手動チェックリスト

スキルを使用できない場合は、コミット前に以下をすべて確認してください：

- [ ] ドキュメント内のすべての事実上の主張を列挙した
- [ ] 各主張に対して以下のいずれかの情報ソースを特定した：
  - [ ] コード・実装ファイル（ファイル名 + 行番号）
  - [ ] ドキュメント・仕様書（ファイル名 + セクション名）
  - [ ] テスト結果・実行ログ（出所と内容）
- [ ] 情報ソースが存在しない主張は削除したか、「根拠なし」と明記したか
- [ ] 架空の情報源（「～パラメータに記載」など実際に確認していない記載）がないか
- [ ] 参照ファイルの行番号は、実際に読み込んで確認した行番号であるか
- [ ] 不確実性のある表現（「～のように思われる」「～だと考えられる」）がないか、または理由を明記しているか

### 方法 3：コミット前の自問自答

コミット直前に、各主張に対して以下を自問してください：

```
Q1: この主張の根拠は何か？
Q2: その根拠は、実装ファイル/ドキュメント/テスト結果に実際に記載されているか？
Q3: ファイル名・行番号は正確か？（実際に読み込んで確認したか？）
Q4: 根拠が確実でない場合、その旨を記載しているか、または削除したか？
```

すべてに「はい」と答えられない場合は、コミットする前に修正してください。

## 本プロジェクトでの適用例

### 誤った例

```markdown
## モード 1: Agent 会話テスト

テスト対象: Bedrock Agent API インボケーション

**参考**: `cfn-templates/bedrock-agent.yaml` Instruction パラメータ
```

**問題**: 
- `cfn-templates/bedrock-agent.yaml` には Instruction パラメータが存在しますが、
- 「ユーザー質問への対応パターン」の根拠となる要件定義書が存在しません
- 実装ファイルを確認せず、根拠なく「参考」を記載

### 修正された例

```markdown
## モード 1: Agent 会話テスト

テスト対象: Bedrock Agent API インボケーション

**実装**: 
- `cfn-templates/bedrock-agent.yaml` 行 16-62: Instruction パラメータ定義
- `lib/lambda_handler.py` 行 36-68: lambda_handler() 6 つのアクション処理

**検証済みの仕様:**
- Agent がユーザー質問を受け取る
- Knowledge Base から関連ランブックを検索する
- Lambda アクション（FR-01 ～ FR-06）を実行する

**未検証の仕様:**
- 各 FR アクションの実行順序（優先度）
  → 根拠となる要件定義書を作成する必要があります
- ユーザー入力パターンごとの詳細な処理フロー
  → 以下のドキュメント作成が必要：
    * ユーザー入力パターン定義書
    * 各パターンに対応する FR 優先度仕様
```

## スキルの場所

- **ファイル**: `.agents/skills/source-verification/SKILL.md`
- **用途**: ドキュメント・仕様書作成時の品質保証

## 関連するドキュメント

- `AGENTS.md`: システムユースケース - 3 つの入力モード（検証済み）
- `docs/requirements.md`: 機能要件定義（検証済み）
- `docs/func_docs/FR-01.md` ～ `FR-06.md`: 個別機能仕様（検証済み）

## よくある質問

**Q: 仕様がまだ決まっていない場合は？**

A: その旨を明記し、**今後定義が必要**と記載してください。架空の仕様で補完してはいけません。

```markdown
## ユーザー入力パターン処理フロー

**現状:** 実装されていますが、パターンの根拠となる仕様ドキュメントが存在しません

**今後必要な作業:**
1. ビジネス要件から「典型的なユーザー入力パターン」を抽出
2. 各パターンに対応する FR アクションの優先度を定義
3. パターン定義書を作成・レビュー
4. テストケースを実装・検証
```

**Q: 参照ファイルが削除された場合は？**

A: git の削除コミットを参照として記載してください。

```markdown
**参考**: 
- 削除前の実装: git commit 089a7a6 で bedrock-agent.yaml の行 47-50 を削除
  （理由: 仕様根拠不明な「ユーザー質問への対応パターン」セクション）
```

---

**最後に**: このガイドに従うことで、プロジェクトのドキュメント品質を保ち、架空の仕様による混乱を防ぐことができます。
