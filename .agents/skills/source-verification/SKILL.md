# ドキュメント・仕様の情報ソース検証スキル

## 1. 概要 (Overview)

このスキルは、ドキュメント、仕様書、設計書、テスト計画などを作成・編集する際に、**すべての主張に対して明確な情報ソース（実装ファイルの行番号、実際のコード、ソースドキュメント）を要求**し、架空の情報源や検証されていない仮定に基づいた記載を防止します。

## 2. 対象（トリガー条件）

* ドキュメント（`.md`, `.txt`）の新規作成・編集時
* 仕様書（`requirements.md`, `specification.md`）の作成・修正時
* テスト計画書、設計書の作成・修正時
* 情報ソースを引用または参照する内容の記載時
* 「～のように思われる」「～に基づいて」といった根拠が不明な主張を記載しようとするとき

## 3. コア・ワークフロー (Core Workflow)

### ステップ 1: 主張の識別

ドキュメント内のすべての**事実上の主張**を識別します：

* 「機能 X は Y を実装している」
* 「CloudFormation テンプレートは Z 行で定義される」
* 「Lambda 関数は FR-01 を処理する」
* 「ユーザー入力パターンは A → B → C の順序で処理される」

### ステップ 2: 情報ソースの確認（必須）

各主張に対して、**必ず以下のいずれかの情報ソースを特定**してください：

#### 2.1 コード・実装ファイル（最優先）

```
例：
- ✅ 「cfn-templates/bedrock-agent.yaml 行 115: Instruction パラメータで Agent プロンプトを定義」
- ✅ 「lib/lambda_handler.py 行 36-68: lambda_handler() 関数が 6 つの FR アクションを分岐処理」
- ✅ 「cfn-templates/eventbridge-alarms.yaml 行 20-56: EC2HighCPUAlarmRule で CloudWatch ALARM をトリガー」
```

#### 2.2 ドキュメント・仕様書

```
例：
- ✅ 「docs/requirements.md: FR-01 ～ FR-06 機能定義」
- ✅ 「AGENTS.md セクション 0: システムユースケース - 3 つの入力モード」
```

#### 2.3 テスト結果・実行ログ

```
例：
- ✅ 「CloudWatch Logs に lambda_handler() の実行ログ記録」
- ✅ 「git commit 089a7a6 で bedrock-agent.yaml の 「ユーザー質問への対応パターン」セクション削除」
```

### ステップ 3: 情報ソースが存在しない場合の対応（重要）

**情報ソースが確認できない主張は、以下のいずれかの対応を取ってください：**

#### ❌ 架空の情報源を作成しない

```
悪い例：
「出所: cfn-templates/bedrock-agent.yaml Instruction パラメータ」
（実際には、パターンの根拠はどのドキュメントにも記載されていない）
```

#### ✅ 正確な表現に修正する

```
良い例 1: 根拠がない場合は明記
「ユーザー質問への対応パターンは、実装されていますが仕様根拠ドキュメントが存在しません」

良い例 2: 仕様を創造する際の要求事項を記載
「以下の仕様定義が必要です：
- ユーザー入力パターン A ～ C の定義
- 各パターンに対応する FR アクションの優先度
- 根拠となる要件定義書の作成」
```

#### ✅ 当該内容を削除する

架空の根拠に基づいた内容を記載する場合は、その内容を**完全に削除**してください。

### ステップ 4: 検証チェックリスト

ドキュメント完成前に、以下をすべてチェックしてください：

- [ ] 各主張に対して明確な情報ソース（ファイル + 行番号、またはドキュメント章節）が存在する
- [ ] 架空の情報源（「～パラメータに記載」など実際に確認していない記載）がない
- [ ] 「～のように思われる」「～だと考えられる」などの不確実性のある表現は、理由を明確にしている
- [ ] 情報ソースが存在しない場合は、その旨を明記し、「今後の仕様定義が必要」と記載している
- [ ] 参照ファイルの行番号は、実際に読み込んで確認した行番号である

## 4. 行動原則 (Rules & Principles)

### 4.1 妥協のない検証

* **「おそらく正しい」は認めない**：主張の根拠が不確実な場合は、その旨を明記するか削除する
* **架空の情報源を作成しない**：ドキュメントには存在しない情報を「出所: ～」と記載しない

### 4.2 透明性の確保

* すべての主張に対して、ユーザーが情報ソースを検証可能な形式で記載する
* 情報ソースが不確実な場合は、その理由を明確に説明する

### 4.3 段階的改善

* 仕様や実装が未決定の部分については、「今後定義が必要」と明記し、具体的なアクションアイテムを提示する
* 架空の仕様で補完しない

## 5. よくある誤りと修正例

### 5.1 誤り例 1：ドキュメント上の記載を「実装ファイルの内容」と混同

```
❌ 悪い例：
「AGENTS.md セクション 12 に 'ユーザー質問への対応パターン'の仕様が記載されている」
（実際には、AGENTS.md の内容は bedrock-agent.yaml から引用されたもので、
 パターンの根拠となるドキュメントが存在しない）

✅ 修正：
「bedrock-agent.yaml 行 47-50 に『ユーザー質問への対応パターン』が記載されていますが、
 このパターンの根拠となるドキュメント（要件定義書、設計書）は存在しません」
```

### 5.2 誤り例 2：想定される動作を「検証済みの実装」として記載

```
❌ 悪い例：
「Agent が正しい FR 優先度で実行される」
（実際には、テストを実行していないため、動作未確認）

✅ 修正：
「Agent が正しい FR 優先度で実行されることを確認するため、以下のテストが必要です：
 1. Bedrock Agent API を呼び出し
 2. CloudWatch Logs で実行ログを確認
 3. Lambda が FR-01 → FR-02 の順で実行されたことを確認」
```

### 5.3 誤り例 3：情報ソースの行番号が実際の行と不一致

```
❌ 悪い例：
「bedrock-agent.yaml 行 100 で Instruction パラメータが定義されている」
（実際には行 16 ～ 62）

✅ 修正：
確実に読み込んだ行番号を記載：
「bedrock-agent.yaml 行 16-62 で Instruction パラメータが定義されている」
```

## 6. チェック用ツール

このスキルを適用する際は、以下のツールを活用してください：

* **読み込み検証**: `read` ツール → ファイルの行番号を確認
* **パターン検索**: `grep` ツール → キーワードの出現位置を確認
* **ファイル検索**: `glob` ツール → 関連ファイルの存在確認

## 7. 適用例

### 例 1：テスト仕様ドキュメントの作成時

```
作成対象: docs/test-plan.md

❌ やってはいけないこと：
「モード 1: Agent 会話テスト
- 情報ソース: AGENTS.md セクション 12
- テスト対象: bedrock-agent.yaml の Instruction パラメータ」

（→ セクション 12 は架空の仕様に基づいていたため、この参照自体が無効）

✅ 正しいやり方：
1. 実際にコードを読む
   - cfn-templates/bedrock-agent.yaml 行 16-62 で Instruction を確認
   - lib/lambda_handler.py 行 36-68 で 6 つのアクション処理を確認

2. 根拠を明記
「モード 1: Agent 会話テスト
- 実装: cfn-templates/bedrock-agent.yaml 行 115 で Instruction パラメータを Agent に設定
- Lambda 処理: lib/lambda_handler.py 行 36-68
- テスト対象: Lambda が 6 つの FR アクション（FR-01～FR-06）を正しく処理するか」

3. 未検証部分を明記
「以下は未テスト：
- Knowledge Base から正しくランブックを取得できるか
- SNS 通知が正常に送信されるか」
```

### 例 2：CloudFormation テンプレート参照ドキュメントの作成時

```
作成対象: docs/cfn-architecture.md

✅ 正しい記載方法：
「EventBridge トリガー設定:
- ファイル: cfn-templates/eventbridge-alarms.yaml
- EC2 High CPU ルール: 行 20-56
  - アラーム名プレフィックス: EC2-HighCPU
  - トリガー条件: ALARM 状態
  - Lambda アクション: bottleneck_investigation (FR-02)
- RDS High CPU ルール: 行 58-94
  - 対応 FR: FR-02」

（各ルール、行番号、プロパティを実装から直接確認）
```

## 8. 復習：このスキルを適用した修正例

本プロジェクトで実際に発生した誤りと修正：

| 誤り | 原因 | 修正方法 |
|------|------|---------|
| `docs/test-specifications-sources.md` に架空の情報源記載 | 実装ファイルを確認せず推測で記載 | ファイル削除 + 根拠となるドキュメント作成は今後 |
| AGENTS.md セクション 12 で未検証の「ユーザー対応パターン」記載 | 実装ファイルの内容確認不足 | セクション削除 + 仕様定義プロセス再構築 |
| bedrock-agent.yaml Instruction パラメータに根拠のないパターン定義 | 仕様要件が明確でなかった | 検証済みの内容のみ保持 + パターン削除 |

---

## 9. 注記

このスキルは、プロジェクトの信頼性と保守性を確保するためのものです。不確実な仕様を検証可能な形で記載し、架空の根拠を排除することで、将来の開発者が安心して参照できるドキュメントを実現します。

何か質問や修正が必要な場合は、コミット前に必ず **「情報ソースは何か」** を自問自答してください。
