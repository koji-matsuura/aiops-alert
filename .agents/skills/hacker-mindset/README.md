# Hacker Mindset Skill - 実装ガイド

## 概要

このスキルは、**Eric S. Raymond の「How To Become A Hacker」を実践的な判断ツールに変換**したものです。

目的：
- 理論的正確性より実践的判断を優先
- エラー対応時の判断を明確化
- 試行錯誤から学ぶプロセスを構築
- 「完遂したい欲望」に流されない判断基準を提供

## 実装の特徴

### 1. 5つのハッカー態度 → 判断基準に変換

| 態度 | 実装 | 判断基準 |
|------|------|---------|
| 問題を解く | 興味を持つ | エラーは「解くべき問題」 |
| 二度解かない | 根本原因を理解 | 同じエラー = 理解不足の証 |
| drudgery は悪 | 試行錯誤は投資 | 修正→テスト→失敗は思考活動 |
| Freedom | 理論より現実 | テスト結果が最優先 |
| 能力が必須 | 常に検証 | エラー = 能力不足を認識 |

### 2. 実践ワークフロー（5フェーズ）

```
Phase 1: 問題検出 → 2: 原因調査 → 3: 詳細分析 → 4: 修正 → 5: 学習記録
```

各フェーズは**意思決定ツリー**として実装され、判断基準が明確化されています。

### 3. 反パターン例（私が犯した誤り）

```
❌ 誤り: 理論を信じ続ける
✅ 正しい: テスト結果を優先
```

**実例から学ぶ**: cfn-lint エラーへの対応ケーススタディ

### 4. 実行可能なチェックリスト

```
□ エラー出現時: Phase 1 チェックリスト
□ 複数回失敗時: 異常系チェックリスト
□ 欲望に流された時: 対抗策チェックリスト
```

---

## 使用シーン

### シーン 1: エラーに直面した時

1. **問題解決チェックリスト** を開く
2. **Phase 1** から開始
3. 各 □ を確認しながら進む
4. **最後に Phase 5** で学習を記録

### シーン 2: 複数回失敗している時

1. **問題解決チェックリスト** → 「異常系」セクション
2. **修正を止める** 判断を確認
3. 原因調査に切り替える

### シーン 3: 修正を続けるべきか迷った時

1. **SKILL.md** の「意思決定ツリー」確認
2. または **問題解決チェックリスト** の「『完遂したい』欲望への対抗策」
3. **テスト結果（現実）** を優先する

### シーン 4: 判断基準を再確認したい時

1. **references/hacker-attitude.md** で各態度の詳細確認
2. チェックリストで自分の判断が正しいか検証

## 主要な判断基準

### 判断基準 1: Linter エラーの解釈

```
cfn-lint エラー = 修正が失敗している（絶対的判定）
  ↓
【即座に判定】修正を止める
  ↓
【対応】原因調査へ
```

### 判断基準 2: 複数回失敗時の対応

```
同じセクションで 2 回以上失敗
  ↓
【警告】根本原因がまだ理解されていない
  ↓
【対応】修正を止める、または戻す
```

### 判断基準 3: 理論 vs 現実

```
「理論は正しいはず」 vs 「テスト結果がエラーを示している」
  ↓
【優先】テスト結果（現実）を選ぶ
  ↓
【対応】理論を疑う、修正を止める
```

### 判断基準 4: ユーザーの修正尊重

```
ユーザーが正しく修正した
  ↓
【確認】テスト成功、インデント正しい
  ↓
【判定】ユーザーの能力を認識、ここで終了
```
