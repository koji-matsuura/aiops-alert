# 問題解決チェックリスト

**目的**: エラーや問題に直面した時、正しいハッカー的判断をするためのチェックリスト

---

## Phase 1: 問題検出時

```
エラーまたは問題が出現した
  ↓
□ 【態度1】「これは解くべき問題」と認識したか？
   └─ NO → 認識を改める、質問を始める

□ 【確認】このエラーの重大性は？
   ├─ デプロイ不可（cfn-lint error）
   │   └─ 【即座に対応】修正を止める
   ├─ 警告（lint warning）
   │   └─ 重要か判定
   └─ 情報（info）
       └─ 後回し OK

□ 【記録】エラーメッセージを完全に記録した？
   └─ エラーテキスト全文をコピー

□ 【初期判定】このエラーは「新しく出現した」か「既存」か？
   └─ 既存 → 原因はわかっているか？
```

**結論**: 修正を止めて原因調査に移る

---

## Phase 2: 原因調査

```
修正を止めた
  ↓
□ 【第1の質問】「このエラーは私の修正が原因か？」

  【検証方法1】git diff で修正を確認
  └─ git diff HEAD cfn-templates/lambda-function.yaml
     └─ 修正内容を全て確認

  【検証方法2】修正前後の違いを理解
  └─ 「何が変わった？」
     ├─ 値？
     ├─ キー名？
     ├─ インデント？
     ├─ 構造？
     └─ 記録

  【判定】
  ├─ YES: 修正が原因 → Phase 3 へ
  ├─ NO: 修正が原因ではない → 別原因を調査
  └─ UNCLEAR: 確信が持てない → さらに比較

□ 【第2の質問】「複数ファイルを修正した場合、どのファイルが原因？」

  【検証方法】ファイルごとにテスト
  └─ cfn-lint cfn-templates/lambda-function.yaml
  └─ cfn-lint cfn-templates/bedrock-agent.yaml
  └─ ... 各ファイル個別にテスト
     └─ エラーが出たファイルを特定

□ 【第3の質問】「修正内容の何が cfn-lint エラーを引き起こしたか？」

  【分析】修正内容を行ごとに確認
  └─ git diff -U0 で変更行のみ表示
  └─ 各変更がエラーと関係するか判定
     ├─ インデント修正 ← よくエラー原因
     ├─ 値の変更
     ├─ キー名の変更
     └─ 構造の変更

□ 【根本原因の仮説】
  ├─ 原因は何か？（明確に記述）
  ├─ なぜこの原因が発生した？
  └─ 修正前後で何が異なった？
```

**結論**: 根本原因が特定される、または「修正が原因ではない」と確定

---

## Phase 3: 根本原因の詳細分析

```
根本原因が仮説された
  ↓
□ 【深掘り】その原因がなぜ発生したのか？

  ├─ 知識不足？
  │  └─ AWS/CloudFormation ドキュメント読む
  │     └─ 該当セクションをコピー → 分析
  │
  ├─ 実装ミス？
  │  └─ コードを手で書き直す
  │     └─ どこが誤っていたか特定
  │
  ├─ 仕様理解不足？
  │  └─ AWS CLI で実際に実行
  │     └─ 動作を確認
  │
  ├─ 前提条件の誤り？
  │  └─ 環境変数、設定値を確認
  │
  └─ 設計の誤り？
     └─ アーキテクチャを再検討

□ 【学習】この原因を二度と引き起こさない方法は？
  ├─ チェックリストに追加？
  ├─ ドキュメント参照？
  ├─ テスト自動化？
  └─ 記録
```

**結論**: 根本原因を「完全に」理解する

---

## Phase 4: 修正実装

```
根本原因が完全に理解された
  ↓
□ 【修正設計】どう修正するか？
  ├─ 複数の方法がある？
  │   └─ 各方法の利点・欠点を比較
  │
  ├─ 最小限の修正は？
  │   └─ 不要な変更を含めない
  │
  └─ 修正で根本原因が解決されるか？
      └─ 修正後の状態をシミュレート

□ 【修正実装】修正を実装
  └─ edit ツール、または手動で修正

□ 【即座に検証】テストツール実行
  ├─ cfn-lint cfn-templates/lambda-function.yaml
  ├─ エラー消滅？ → YES / NO
  ├─ 新しいエラー発生？ → YES / NO
  └─ 既存機能に影響？ → YES / NO

  【判定】
  ├─ エラー消滅 + 新エラーなし → Phase 5 へ
  ├─ エラー残存 → 修正が不完全 → Phase 3 に戻る
  └─ 新エラー発生 → 別の問題発生 → Phase 1 に戻る
```

**結論**: 修正が成功する、または失敗して原因調査に戻る

---

## Phase 5: 学習記録

```
問題が解決された
  ↓
□ 【記録】同じエラーは二度と繰り返さない
  ├─ 原因は何か？（明確に記述）
  │   └─ 例: インデント不一致により cfn-lint YAML syntax error
  │
  ├─ 解決策は何か？（手順を記述）
  │   └─ 例: git diff で修正内容を確認、インデント 2 スペース統一
  │
  ├─ なぜこの誤りが起きた？（学習）
  │   └─ 例: edit ツール使用時、複数置換で不正確な oldString を指定
  │
  └─ 次回の予防策は？
      └─ 例: 修正前に必ず git show で元のテキスト確認

□ 【同じパターン確認】他の箇所に同じ誤りがないか？
  ├─ grep で検索
  │   └─ 同じパターンを検出
  │
  ├─ 検出されたら同じ修正を適用
  │
  └─ 検出されなかったら OK

□ 【記録完了】
  └─ このチェックリストの該当セクションに「学習済み」マーク
```

---

## 異常系: 複数回失敗した場合

```
同じセクションで 2 回以上修正失敗
  ↓
□ 【警告】何かが根本的に間違っている

  ├─ git log で修正履歴を確認
  │   └─ 何回修正を試みた？ → 記録
  │
  ├─ git diff HEAD~X HEAD で全修正を確認
  │   └─ 修正の履歴から何が起きているか分析
  │
  └─ 【判定】
      ├─ 修正が完全に誤っている → Phase 3 に戻る
      ├─ アプローチが誤っている → 代替案を検討
      └─ 仕様理解が誤っている → ドキュメント再読

□ 【決定】修正を一度全部戻すべきか？
  ├─ YES → git checkout で元に戻す
  │   └─ git checkout HEAD cfn-templates/lambda-function.yaml
  │   └─ Phase 2 に戻る
  │
  └─ NO → 代替アプローチで続行

□ 【重要】「完遂したい」という欲望に流されていないか？
  ├─ エラーが出ている
  ├─ 複数回修正が失敗している
  ├─ 修正を止めるべき時期
  └─ 修正を止める ← ここが重要
```

---

## 「完遂したい」欲望への対抗策

```
状況: 修正を完成させたいという感情が強い
  ↓
□ 【危険信号確認】
  ├─ cfn-lint エラーが出ているか？
  │   └─ YES → 完成していない
  │
  ├─ 複数回修正が失敗しているか？
  │   └─ YES → 根本原因が理解されていない
  │
  ├─ ユーザーが正しく修正しているか？
  │   └─ YES → 自分より能力が上、尊重する
  │
  └─ 「理論は正しいはず」と信じ続けていないか？
      └─ YES → 理論を捨てて現実を優先

□ 【対応】修正を止める
  └─ 原因調査に切り替える
     └─ テスト結果（現実）を完全に信じる
```

---

## チェック完了

```
全フェーズを実行した
  ↓
□ 問題が完全に解決されたか？
  ├─ YES → 完了
  │   └─ 学習内容を記録
  │
  └─ NO → 上記フェーズを繰り返す
      └─ または別アプローチを検討
```

---

## 使用方法

1. **エラーが出た時**
   - このチェックリストを開く
   - Phase 1 から開始
   - 各 □ を確認しながら進む

2. **複数回失敗した時**
   - 「異常系」セクションを確認
   - 修正を止めるべき時期を認識

3. **修正を続けるべきか判定に迷った時**
   - 「『完遂したい』欲望への対抗策」セクション参照
   - 現実（テスト結果）を優先
