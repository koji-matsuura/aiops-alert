# 実例: エラー対応の正しい判断基準 - 複数技術領域

**対象**: CloudFormation（cfn-lint）、Python（pytest）、Go（go test）、API

**共通パターン**: 修正後のテスト実行でエラーが出現した場合、正しい対応と誤った対応の対比

---

## シナリオ別ケース

### ケース 1: CloudFormation (cfn-lint エラー)

```
目標: IAM ロール命名を統一する
  ↓
修正内容: lambda-function.yaml のロール名を変更
  ↓
実行: cfn-lint cfn-templates/lambda-function.yaml
  ↓
結果: エラー出現 → 対応方法は？
```

### ケース 2: Python (pytest 失敗)

```
目標: Lambda ハンドラーを修正する
  ↓
修正内容: lib/lambda_handler.py でイベント処理を変更
  ↓
実行: pytest tests/test_lambda_handler.py
  ↓
結果: テスト失敗 → 対応方法は？
```

### ケース 3: Go (go test 失敗)

```
目標: API ハンドラーにバリデーションを追加
  ↓
修正内容: handler.go でリクエスト検証ロジック追加
  ↓
実行: go test ./...
  ↓
結果: テスト失敗 → 対応方法は？
```

### ケース 4: API (リクエスト失敗)

```
目標: API のエンドポイントを変更
  ↓
修正内容: 認証ヘッダのフォーマット変更
  ↓
実行: curl コマンドでテスト
  ↓
結果: 401 Unauthorized → 対応方法は？
```

---

## 【誤った対応】（すべての技術領域で共通するパターン）

### ステップ 1: テスト結果を軽視

```
テスト実行でエラー出現
  ↓
【判定】「これは修正の主要部分とは無関係」
  ↓
修正を続ける ← 誤り
```

**理由**：
- 「正しい設計」という信念に基づいて判定
- 現実のテスト結果を軽視
- 理論を優先

**ツール別の誤った判定**：
- **CloudFormation**: 「cfn-lint が厳格すぎる」
- **Python**: 「pytest が完全でない」
- **Go**: 「go test が厳しすぎる」
- **API**: 「サーバーが厳格」

### ステップ 2: テスト失敗が増える

```
修正2 → テスト失敗
修正3 → テスト失敗
修正4 → テスト失敗
...
↑ テスト失敗が増えている
```

**本来すべき判定**：「何かが根本的に間違っている」

**実際の判定**：「次の修正で解決するはず」

### ステップ 3: ユーザーが正しく修正

```
ユーザーが手動で正しく修正
  ↓
テスト実行 → 成功
  ↓
コード確認 → 実装正しい
```

**本来すべき判定**：「ユーザーの修正を尊重する、ここで終了」

**実際の判定**：「さらに修正が必要かもしれない」

---

## 【正しい対応】（ハッカー的判断 - すべての技術領域で共通）

### ステップ 1: テスト結果を直視する

```
テスト実行でエラー出現
  ↓
【判定】「テストがエラーを示している = 実装が失敗している」
         （理論ではなく現実を優先）
  ↓
修正を止める
```

**判断基準**（ツール別）：
- **CloudFormation**: cfn-lint エラー = 実装失敗
- **Python**: pytest 失敗 = 実装失敗
- **Go**: go test 失敗 = 実装失敗
- **API**: HTTP 4xx/5xx = リクエスト失敗

**共通原則**: テスト失敗は「修正が失敗している」の確定的証拠

### ステップ 2: 根本原因を調査

```
修正を止めた
  ↓
差分を確認（git diff またはファイル直接確認）
  ↓
【質問1】「テスト失敗と修正内容の関係は？」
  ├─ 直接関係ある → 修正を見直す
  ├─ 間接的関係？ → さらに調査
  └─ 無関係？ → 他の要因を探す

【質問2】「複数ファイルを修正した場合、どの修正が原因？」
  ├─ 各ファイルを個別にテスト
  └─ テスト失敗するファイルを特定

【質問3】「修正前後で何が変わった？」
  ├─ 値の変更？
  ├─ ロジック変更？
  ├─ インデント変更？
  └─ 【比較】どれがテスト失敗を引き起こした？

ツール別の詳細確認：
  ├─ CloudFormation: キー名、型、インデント確認
  ├─ Python: 構文、型、参照の確認
  ├─ Go: 型、関数シグネチャの確認
  └─ API: リクエスト形式、ヘッダの確認
```

### ステップ 3: テスト失敗が増えたら即座に気づく

```
修正1 → テスト失敗
  ↓
修正2 → テスト失敗
  ↓
【警告】同じセクションで複数回テスト失敗
  ↓
【判定】「何かが根本的に間違っている」
  ↓
【対応】
  ├─ これまでの修正を全て見直す
  ├─ 修正を一度全て戻す
  ├─ 最初から開始
  └─ または別アプローチを検討
```

### ステップ 4: ユーザーの修正を尊重

```
ユーザーが正しく修正した
  ↓
【検証】
  ├─ テスト実行 → 成功か失敗か確認
  ├─ コード確認 → 実装は正しいか
  ├─ 修正内容の意図を理解できるか？
  │   └─ YES → 修正は正しい
  │
  └─ ユーザーの能力を評価
      └─ ユーザー > 自分の能力を認識

【判定】「ここで終了、ユーザーの判断を信じる」
```

## 学習ポイント

### 何を学ぶべきだったのか

1. **テスト失敗 = 絶対的判定**（すべてのツール共通）
   - エラー出現 = 実装が失敗している、この判定に例外なし
   - cfn-lint でも、pytest でも、go test でも、API テストでも同じ
   
2. **理論と現実の優先順位**
   - 理論が正しいと感じても、テスト結果が否定したら、理論を捨てる
   - 「設計は正しい」≠「実装は成功している」
   
3. **試行錯誤の本質**
   - 「修正 → テスト → 失敗 → 修正」は思考活動、必要な投資
   - その中で「何が間違っていたのか」を理解する
   
4. **同じテスト失敗が複数回出現した時の対応**
   - 根本原因がまだ理解されていない
   - 修正を止めて原因を徹底調査する必要がある
   
5. **ユーザーの判断を尊重する**
   - ユーザーが正しく修正した後に「さらに修正が必要？」と疑わない
   - テストで検証して正しければ、そこで終了

### 次に同じ状況が起きた時（すべての技術領域）

```
テスト失敗出現
  ↓
【即座に判定】実装が失敗している
  ↓
【対応】修正を止める → 原因調査
  ↓
1 回目のテスト失敗で根本原因を理解する
  ↓
同じテスト失敗は二度と発生させない
```

## パターン比較表

| 技術領域 | テスト失敗の意味 | 誤った対応 | 正しい対応 |
|---------|-----------------|----------|---------|
| **CloudFormation** | スキーマ違反 | cfn-lint が厳格と思う | cfn-lint エラー = 実装失敗 |
| **Python** | 処理失敗 | pytest が完全でないと思う | pytest 失敗 = 実装失敗 |
| **Go** | コンパイル/ロジック失敗 | go test が厳格と思う | go test 失敗 = 実装失敗 |
| **API** | リクエスト/レスポンス失敗 | サーバが厳格と思う | HTTP エラー = リクエスト失敗 |

**共通結論**: テスト失敗 = 実装失敗（どの技術領域でも同じ原則）
