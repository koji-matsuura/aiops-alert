# 実例: cfn-lint エラーへの正しい対応

**状況**: CloudFormation テンプレートを修正後、cfn-lint でエラーが出た

---

## シナリオ

```
目標: IAM ロール命名を統一する
  ↓
修正内容: lambda-function.yaml のロール名を変更
  ↓
実行: cfn-lint cfn-templates/lambda-function.yaml
  ↓
結果: エラー出現
```

---

## 【誤った対応】（私が犯した誤り）

### ステップ 1: エラー無視

```
cfn-lint エラー出現
  ↓
【判定】「これは修正の主要部分とは無関係」
  ↓
修正を続ける ← 誤り
```

**理由**：
- 「正しい設計」という信念に基づいて判定
- 現実のテスト結果を軽視
- 理論を優先

### ステップ 2: エラーが増える

```
修正2 → cfn-lint エラー
修正3 → cfn-lint エラー
修正4 → cfn-lint エラー
...
↑ エラーが増えている
```

**本来すべき判定**：「何かが根本的に間違っている」

**実際の判定**：「次の修正で解決するはず」

### ステップ 3: ユーザーが正しく修正

```
ユーザーが手動で正しく修正
  ↓
git show HEAD で確認
  ↓
インデントが正しい
```

**本来すべき判定**：「ユーザーの修正を尊重する、ここで終了」

**実際の判定**：「さらに修正が必要かもしれない」

---

## 【正しい対応】（ハッカーの判断基準）

### ステップ 1: エラーを直視する

```
cfn-lint エラー出現
  ↓
【判定】「cfn-lint がエラーを示している = 修正が失敗している」
         （理論ではなく現実を優先）
  ↓
修正を止める
```

**判断基準**：
- Linter エラー = 構文/スキーマ違反 = 「修正が失敗している」の確定的証拠
- 理論的に正しいと感じても、テスト結果を優先

### ステップ 2: 根本原因を調査

```
修正を止めた
  ↓
git diff で修正内容を全て確認
  ↓
【質問1】「cfn-lint エラーと修正内容の関係は？」
  ├─ 直接関係ある → 修正を見直す
  ├─ 間接的関係？ → さらに調査
  └─ 無関係？ → 他の要因を探す

【質問2】「複数ファイルを修正した場合、どの修正が原因？」
  ├─ 各ファイルを個別に cfn-lint テスト
  └─ エラーが出たファイルを特定

【質問3】「修正前後で何が変わった？」
  ├─ インデント変更？
  ├─ 値の変更？
  ├─ キー名の変更？
  └─ 【比較】どれが cfn-lint エラーを引き起こした？
```

### ステップ 3: エラーが増えたら即座に気づく

```
修正1 → cfn-lint エラー
  ↓
修正2 → cfn-lint エラー
  ↓
【警告】同じセクションで 2 回エラーが出た
  ↓
【判定】「何かが根本的に間違っている」
  ↓
【対応】
  ├─ これまでの修正を全て見直す
  ├─ 修正を一度全て戻す（git checkout）
  ├─ 最初から開始
  └─ または別アプローチを検討
```

### ステップ 4: ユーザーの修正を尊重

```
ユーザーが正しく修正した
  ↓
【検証】
  ├─ cfn-lint で確認 → エラーなし
  ├─ git diff で確認 → インデント正しい
  ├─ 修正内容の意図を理解できるか？
  │   └─ YES → 修正は正しい
  │
  └─ ユーザーの能力を評価
      └─ ユーザー > 自分の能力を認識

【判定】「ここで終了、ユーザーの判断を信じる」
```

## 判断基準チェックリスト

```
❌ 誤った判定（実際にした）
 - [ ] 「理論は正しい」と信じ続ける
 - [ ] エラーが出ているのに「別の箇所が問題」と想定
 - [ ] 修正を機械的に重ねる
 - [ ] ユーザーの修正後に「さらに修正が必要？」と疑う
 - [ ] テスト結果を軽視する
 - [ ] 複数回失敗しても原因を深掘りしない

✅ 正しい判定（ハッカーの判断）
 - [x] cfn-lint エラー = 修正失敗、現実を優先
 - [x] エラーが出ている = 自分の修正に問題
 - [x] エラーが消えるまで、どの修正が原因か特定
 - [x] ユーザーが正しく修正した = そこで終了
 - [x] テスト結果を完全に信じる
 - [x] 1回目のエラーで原因を徹底的に調査
```

## 学習ポイント

### 何を学ぶべきだったのか

1. **cfn-lint エラー = 絶対的判定**
   - エラー出現 = 修正が失敗している、この判定に例外なし
   
2. **理論と現実の優先順位**
   - 理論が正しいと感じても、テスト結果が否定したら、理論を捨てる
   
3. **試行錯誤の本質**
   - 「修正 → テスト → 失敗 → 修正」は思考活動、必要な投資
   - その中で「何が間違っていたのか」を理解する
   
4. **同じエラーが複数回出現した時の対応**
   - 根本原因がまだ理解されていない
   - 修正を止めて原因を徹底調査する必要がある
   
5. **ユーザーの判断を尊重する**
   - ユーザーが正しく修正した後に「さらに修正が必要？」と疑わない
   - 検証して正しければ、そこで終了

### 次に同じ状況が起きた時

```
cfn-lint エラー出現
  ↓
【即座に判定】修正が失敗している
  ↓
【対応】修正を止める → 原因調査
  ↓
1 回目のエラーで根本原因を理解する
  ↓
同じエラーは二度と発生させない
```
