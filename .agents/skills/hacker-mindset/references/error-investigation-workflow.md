# エラー調査ワークフロー - 汎用版

**適用対象**: CloudFormation、プログラム（Python/Go/JS）、API、インフラストラクチャ全般  
**目的**: エラーが出現した時の標準化された調査手順を提供

---

## 概要

エラーが出現した際、**同じセクションで複数回失敗することを防ぐ** ために、根本原因を初回で完全に理解するための調査ワークフロー。

**原則**: No problem should ever have to be solved twice.

---

## ワークフロー

### Phase 0: エラー情報の収集

```
エラー出現
  ↓
□ エラーメッセージ全文をコピー
  └─ メモ、ノート、チケットに記録

□ エラーコンテキストを記録
  ├─ どのツール/サービスが出力したか？
  │   ├─ cfn-lint
  │   ├─ pytest
  │   ├─ Go test
  │   ├─ Node.js runtime
  │   ├─ API エラー
  │   └─ その他
  │
  ├─ エラーレベルは？
  │   ├─ 実行不可（Error）
  │   ├─ 警告（Warning）
  │   └─ 情報（Info）
  │
  ├─ 発生時刻は？
  ├─ 再現可能か？
  └─ 初回出現か、既出か？

□ 修正履歴を確認
  ├─ git log -5 で直近修正確認
  ├─ git diff で修正内容確認
  └─ タイムスタンプと エラー出現時刻の関連性
```

---

### Phase 1: 修正と エラーの因果関係確認

```
エラー出現 + 修正履歴確認
  ↓
【質問1】「このエラーは私の修正が原因か？」

【確認方法】ツール別ガイド
  
  ┌─ CloudFormation の場合
  │  └─ git diff cfn-templates/*.yaml
  │     └─ 修正がどのファイルに加わったか確認
  │
  ├─ Python コードの場合
  │  └─ git diff lib/
  │     └─ 構文エラー、インデント、型の誤りを確認
  │
  ├─ Go コードの場合
  │  └─ git diff ./*.go
  │     └─ パッケージ名、型宣言、関数シグネチャを確認
  │
  ├─ API レスポンスの場合
  │  └─ curl でリクエストを再現
  │     └─ リクエストヘッダ、ボディが正しいか確認
  │
  └─ インフラ（EC2/RDS）の場合
     └─ CloudWatch ログから時系列を追跡
        └─ 修正とエラー出現時刻の相関を確認

【判定】
  ├─ YES: 修正が原因 → Phase 2 へ
  ├─ NO: 修正が原因ではない → 別途調査
  └─ UNCLEAR: 因果関係不明 → さらに詳細分析へ
```

---

### Phase 2: 修正内容の詳細分析

```
修正が原因と判定された
  ↓
【質問2】「修正の何が問題か？」

ツール別の誤りパターン

┌─ CloudFormation の場合
│  ├─ インデント不一致
│  │   └─ git diff -U0 で変更行のみ表示
│  │   └─ スペース 2 個の統一性確認
│  │
│  ├─ ロール名の不一致
│  │   └─ Resource ARN パターンと実際のロール名を比較
│  │
│  ├─ パラメータの型誤り
│  │   └─ Type が String/Number/List 正しいか確認
│  │
│  └─ 仕様理解の誤り
│      └─ AWS CloudFormation ドキュメント参照
│
├─ Python コードの場合
│  ├─ 構文エラー
│  │   └─ コンパイルして行番号を特定
│  │
│  ├─ インデントエラー
│  │   └─ 関数、クラス、ブロック内のインデント確認
│  │
│  ├─ 型エラー
│  │   └─ 変数の型とメソッド呼び出しの一致確認
│  │
│  ├─ 参照エラー（NameError, AttributeError）
│  │   └─ 変数定義位置と参照位置を追跡
│  │
│  └─ ロジックエラー
│      └─ 条件式、ループの誤り確認
│
├─ Go コードの場合
│  ├─ コンパイルエラー（型、宣言）
│  │   └─ go build で詳細なエラーメッセージ確認
│  │
│  ├─ パッケージインポートエラー
│  │   └─ import パス、バージョンの確認
│  │
│  ├─ 関数シグネチャ不一致
│  │   └─ 呼び出し元と定義の引数、戻り値を比較
│  │
│  └─ ポインタ関連
│      └─ & と * の使用が正しいか確認
│
├─ API の場合
│  ├─ ステータスコード 4xx（クライアントエラー）
│  │   └─ リクエスト形式、認証情報確認
│  │
│  ├─ ステータスコード 5xx（サーバエラー）
│  │   └─ サーバログ、API ドキュメント確認
│  │
│  ├─ タイムアウト
│  │   └─ ネットワーク、サーバ負荷、クエリ時間確認
│  │
│  └─ レスポンス形式不一致
│      └─ 期待される JSON/XML スキーマと比較
│
└─ インフラの場合
   ├─ EC2 インスタンス起動失敗
   │   └─ IAM 権限、セキュリティグループ、ストレージ確認
   │
   ├─ RDS 接続エラー
   │   └─ セキュリティグループ、パスワード、ネットワーク確認
   │
   ├─ デプロイスクリプト失敗
   │   └─ 権限、パス、環境変数確認
   │
   └─ ネットワーク接続失敗
       └─ ルート、セキュリティグループ、DNS 確認
```

---

### Phase 3: 根本原因の特定

```
修正内容の詳細分析終了
  ↓
【根本原因を記述】

例 1（CloudFormation）:
  原因: cfn-pipeline.yml の Resource ARN パターンが、
        lambda-function.yaml で生成される実際のロール名と一致していない
  理由: ロール名を「AiopsLambdaExecutionRole」から
        「dev-aiops-lambda-execution-role」に変更したが、
        cfn-pipeline.yml の Resource パターンを更新していなかった

例 2（Python）:
  原因: event['parameters'] が辞書型を返すが、
        loop で値を直接アクセスしようとしていた
  理由: Lambda messageVersion 1.0 仕様を理解していなかった

例 3（API）:
  原因: Authorization ヘッダの形式が誤っていた
  理由: Bearer トークンの前にスペースが入っていなかった

【分析】なぜこの誤りが発生したのか

ツール別の原因分類:
  ├─ 知識不足
  │   └─ ドキュメント、仕様書を読むことで解決
  │
  ├─ 実装ミス
  │   └─ コードをもう一度見直す、テストを追加
  │
  ├─ 設計ミス
  │   └─ アーキテクチャ、API 設計を再検討
  │
  ├─ 前提条件の誤り
  │   └─ 環境変数、設定値、初期状態を確認
  │
  ├─ テスト不足
  │   └─ ユニットテスト、統合テストを追加
  │
  └─ 仕様理解の誤り
      └─ AWS/言語/フレームワークの仕様確認
```

---

### Phase 4: 修正戦略の決定

```
根本原因が特定された
  ↓
【複数の修正方法を検討】

例 1: 単純な修正
  原因: インデント不一致
  修正方法: インデント統一（スペース 2 個）
  検証方法: cfn-lint で エラー消滅確認
  リスク: 低

例 2: 設計修正が必要
  原因: API 設計の誤り
  修正方法: API 仕様を再定義 → 実装修正 → テスト
  検証方法: ユニットテスト + 統合テスト
  リスク: 中～高

例 3: 複数ファイル修正
  原因: 複数箇所で同じパターンの誤り
  修正方法: 全て同時に修正
  検証方法: grep で同じパターン検索 → 全部修正
  リスク: 中（漏れる可能性）

【修正方法の選択】
  ├─ 最小限の修正か？（余計な変更なし）
  ├─ 根本原因を解決するか？（症状対応ではなく）
  └─ 他の箇所に同じパターンがないか？（再発防止）
```

---

### Phase 5: 修正実装と検証

```
修正戦略が決定された
  ↓
□ 修正を実装

□ 即座に検証
  ├─ ツール実行（cfn-lint, pytest, go test など）
  ├─ エラー消滅？
  ├─ 新しいエラー発生？
  ├─ 既存機能に影響？
  └─ 【判定】全て OK か？

【判定結果】
  ├─ 成功: エラー消滅 + 新エラーなし → Phase 6 へ
  ├─ 失敗: エラー残存 → 修正が不完全、Phase 2 に戻る
  └─ 部分失敗: 新エラー発生 → 別の問題、Phase 1 に戻る
```

---

### Phase 6: 学習記録と再発防止

```
問題が完全に解決された
  ↓
□ 根本原因レポート作成
  ├─ 原因は何か？（明確に）
  ├─ なぜ発生したのか？（分析）
  ├─ 解決策は何か？（実装内容）
  ├─ 検証方法は？（テスト方法）
  └─ 次回の予防策は？（ガイドライン）

□ 同じパターンの他の箇所確認
  ├─ grep で検索（CloudFormation なら cfn-templates/*.yaml）
  ├─ grep で検索（Python コードなら lib/*.py）
  ├─ 同じパターン検出？
  └─ 検出されたら同じ修正を適用

□ テスト強化
  ├─ 同じエラーは二度と発生させないテスト追加
  ├─ 単体テスト追加
  └─ 統合テスト追加

□ ドキュメント更新
  ├─ チェックリストに追加
  ├─ トラブルシューティングガイド更新
  └─ チームナレッジベースに記録
```

## 異常系: 複数回失敗した場合

```
同じ箇所で 2 回以上修正失敗
  ↓
【警告】根本原因がまだ理解されていない

【対応】
  ├─ git log で修正履歴確認
  │   └─ 何回修正を試みた？
  │
  ├─ git diff HEAD~X HEAD で全修正を確認
  │   └─ 修正の履歴から何が起きているか分析
  │
  ├─ 【判定】修正方向は正しいか？
  │   ├─ YES: 修正は正しい方向だが不完全
  │   │   └─ 修正を続行（次のフェーズへ）
  │   │
  │   └─ NO: 修正方向が誤っている
  │       └─ 修正を戻す（git checkout）
  │       └─ 代替アプローチを検討
  │       └─ Phase 2 から再開
  │
  └─ 【重要】修正を止めるべき時期を認識
      └─ エラーが増えている → 修正を止める
      └─ 同じセクションで 3 回以上失敗 → 基本設計を再検討
```

## ツール別の調査チェックリスト

### CloudFormation (cfn-lint)

```
□ エラーメッセージをコピー
□ どのファイルのどの行か確認
□ git diff cfn-templates/ で修正内容確認
□ インデント、型、キー名を確認
□ Resource ARN パターンとロール名の一致確認
□ 修正を最小限に
□ cfn-lint で検証
□ 新エラー発生確認
□ 他の cfn-templates/*.yaml ファイルで同じパターン検索
```

### Python (pytest / 実行時エラー)

```
□ エラーメッセージ、スタックトレース全文をコピー
□ エラー発生行番号を確認
□ git diff lib/ で修正内容確認
□ 構文、インデント、型を確認
□ 関数定義と呼び出し側の一致確認
□ 修正を実装
□ pytest で検証（テスト成功か）
□ 実行して検証
□ 他のファイルで同じパターン検索
□ テストケース追加（再発防止）
```

### Go (go build / go test)

```
□ コンパイルエラーメッセージをコピー
□ エラー行番号を確認
□ git diff ./*.go で修正内容確認
□ パッケージ、型、関数シグネチャを確認
□ import パス、バージョンを確認
□ 修正を実装
□ go build で検証
□ go test で検証
□ 他のファイルで同じパターン検索
```

### API (リクエスト/レスポンスエラー)

```
□ HTTP ステータスコードを記録
□ エラーレスポンス全文をコピー
□ curl / Postman で再現
□ リクエストヘッダ、ボディ、クエリパラメータを確認
□ API ドキュメント確認
□ 修正を実装
□ curl で検証
□ ステータスコード、レスポンス形式を確認
□ 他の API エンドポイント確認
```

## 判断基準

```
エラーに直面した時:
  ├─ 修正と エラーの因果関係 = 95% 以上確定したか？
  │   ├─ NO → Phase 1 をもう一度
  │   └─ YES → Phase 2 へ
  │
  ├─ 根本原因を「完全に」理解したか？
  │   ├─ NO → Phase 2,3 をもう一度
  │   └─ YES → Phase 4 へ
  │
  ├─ 修正が最小限か（余計な変更なし）？
  │   ├─ NO → 修正を見直す
  │   └─ YES → Phase 5 へ
  │
  └─ テスト結果が全て OK か？
      ├─ NO → Phase 4 に戻る
      └─ YES → Phase 6 へ
```
