# 5つの基本的なハッカー態度 - 詳細版

**出典**: Eric S. Raymond, "How To Become A Hacker"  
**目的**: 各態度を実装可能な判断基準に変換

---

## 1. The world is full of fascinating problems waiting to be solved

### 原文の意図

ハッカーはプログラムやシステムの問題に対して「退屈な障害」ではなく「解くべき問題」として捉える。

### 実装レベル

```
エラーが出現した
  ↓
【判定】これは「何か」？
  ├─ 理解できない現象？ → 【好奇心を持つ】なぜ起きているのか？
  ├─ 予期しない結果？ → 【関心を持つ】何が変わった？
  └─ 設計の欠陥？ → 【興味を持つ】どう修正する？
```

### 反パターン

```
❌ cfn-lint エラー出現
   ↓
   【判定】「これは処理すべき雑務」と考える
   ↓
   修正を機械的に重ねる（思考停止）
   
✅ cfn-lint エラー出現
   ↓
   【判定】「これは解くべき問題」と考える
   ↓
   「なぜ cfn-lint がこのエラーを出したのか？」と問う
   ↓
   CloudFormation の仕様を調べる
   ↓
   問題の本質を理解する
```

### 判断基準チェックリスト

- [ ] エラーが出現した時、最初に「退屈だ」と感じたか？
- [ ] 「なぜ？」と 3 回以上問いかけたか？
- [ ] 根本原因を理解するまで続けたか？
- [ ] 同じエラーが再度出現する可能性を検討したか？

## 2. No problem should ever have to be solved twice

### 原文の意図

エラーが出現したら、その根本原因を **完全に** 理解してから修正を実装する。そうしなければ、同じエラーが再度出現する。

### 実装レベル

```
エラー 1 回目出現
  ↓
【投資】十分に時間をかけて調査
  ├─ なぜこのエラーが出ているのか？
  ├─ 修正前後で何が変わった？
  ├─ 同じパターンの他の箇所はないか？
  └─ 【結論】根本原因を「完全に」理解した

【記録】
  ├─ 原因は何か？
  ├─ 解決策は何か？
  ├─ 次回はどう防ぐか？
  └─ 同じ誤りが他の箇所にないか確認

エラー 2 回目以降
  ↓
【即座に対応】記録された知識から解決策を適用
```

### 反パターン

```
❌ 悪い例（私が犯した誤り）：

cfn-lint エラー 1 回目
  ↓
【投資なし】適当に修正してみる
  ↓
cfn-lint エラー 2 回目
  ↓
【投資なし】別の修正をしてみる
  ↓
cfn-lint エラー 3 回目以上 ← この時点で異常
  ↓
【判定】「この修正は完成するはず」と信じ続ける ← 誤り
```

### 判断基準チェックリスト

- [ ] 同じエラーが 2 回以上出現したか？
- [ ] 出現した場合、何が原因だったか説明できるか？
- [ ] その原因は「完全に」排除されたか？
- [ ] 同じパターンの他の場所を確認したか？

## 3. Boredom and drudgery are evil

### 原文の意図

退屈で繰り返される作業は「時間の無駄」であり、**自動化かスキップの対象**。

しかし、試行錯誤や問題の深掘りは「思考活動」であり、drudgery ではない。

### 実装レベル

```
作業を見直す
  ↓
【判定】これは何か？
  ├─ 【思考活動】試行錯誤、問題調査、学習
  │   └─ ✅ 必要な投資、続ける
  │
  ├─ 【機械的作業】同じ手順の繰り返し
  │   └─ ❌ drudgery、自動化対象
  │
  └─ 【不必要な作業】目標と無関係な行動
      └─ ❌ 完全に中止
```

### 反パターン

```
❌ 誤り：
「修正 → テスト → 失敗 → 修正...」は退屈だから避ける
  ↓
修正を 1 回だけしたら検証を飛ばす
  ↓
ユーザーが後で正しく修正する

✅ 正しい：
「修正 → テスト → 失敗 → 修正...」は思考活動
  ↓
何度でも繰り返す
  ↓
完全に理解された状態に到達
```

### 判断基準チェックリスト

- [ ] 今している作業は「思考が必要」か、それとも「機械的」か判定したか？
- [ ] 同じテストを何度も繰り返していないか？（→ 自動化対象）
- [ ] エラーが出ているのに、検証を飛ばしていないか？

## 4. Freedom is good

### 原文の意図

権威（理論、公式指針、他者の判定）に盲従しない。

**特に重要**：理論が正しいと感じても、テスト結果（現実）が否定したら、理論を捨てる。

### 実装レベル

```
「正しいはず」という判定
  ↓
【常に検証する】実際にテストを実行
  ├─ テスト成功
  │   └─ ✅ 理論が正しかった、進む
  │
  └─ テスト失敗（エラー出現）
      └─ ❌ 理論が間違っていた、または実装が誤っている
          ↓
          【即座に判定】理論ではなく現実を優先
          ↓
          修正を止める
          ↓
          原因調査に切り替える
```

### 反パターン（私が犯した誤り）

```
❌ 誤り：
「IAM ロール命名統一は正しい設計」（理論）
  ↓
cfn-lint エラー出現
  ↓
【判定】「理論は正しい、別の箇所が問題」と信じ続ける
  ↓
修正を重ねる（権威への盲従）
  ↓
エラーが増える

✅ 正しい：
「IAM ロール命名統一は正しい設計」（理論）
  ↓
cfn-lint エラー出現
  ↓
【判定】「現実がエラーを示している、理論を疑う」
  ↓
修正を止める
  ↓
「なぜエラーが出ているのか？」と調査
  ↓
「理論自体が間違っているのか？」または「実装が誤っているのか？」を判定
```

### 判断基準チェックリスト

- [ ] 「正しいはず」という感覚に陥っていないか？
- [ ] テスト結果を無視していないか？
- [ ] エラーが出ているのに理論を信じ続けていないか？
- [ ] 「理論」と「現実」が一致しない場合、どちらを優先したか？

## 5. Attitude is no substitute for competence

### 原文の意図

「正しい態度」を持つことと「正しく実装する能力」は別である。

態度だけでは問題は解決しない。

### 実装レベル

```
自分の修正に「自信がある」と感じる
  ↓
【必ず検証する】テストツール（cfn-lint など）を実行
  ├─ エラーなし
  │   └─ ✅ OK、次へ
  │
  └─ エラーあり
      └─ ❌ 「自信」は誤りだった、能力不足
          ↓
          【原因】
          ├─ 知識不足？ → ドキュメント読む
          ├─ 実装ミス？ → コード見直す
          ├─ 仕様理解不足？ → 調査
          └─ 【対応】能力を高める
```

### 反パターン

```
❌ 誤り：
「正しい態度で修正している」（自己評価）
  ↓
テストを実行しない、または軽視
  ↓
エラーが出ている事実を無視
  ↓
修正を重ね続ける

✅ 正しい：
「正しい態度で修正しているはず」（仮説）
  ↓
【必ず検証】テスト実行
  ↓
エラー出現
  ↓
【判定】態度は正しかったが能力が不足している
  ↓
能力を高める（学習、調査、理解）
```

### 判断基準チェックリスト

- [ ] 「正しい」という自信を持っているか？
- [ ] その自信を検証したか？（テスト実行）
- [ ] テスト結果がエラーなら、「自信が誤り」と認識したか？
- [ ] 「完成させたい」という欲望と「能力」を区別したか？

## 統合判定フロー

```
問題またはエラー発生
  ↓
【態度1: 問題を「解くべき対象」と見なす】
  ├─ NO → スキルを向上させる必要
  └─ YES → 進む
  
【態度2: 根本原因を完全に理解する】
  ├─ 同じエラーが複数回出現している？
  │   └─ YES → 根本原因がまだ理解されていない、深掘り
  └─ 進む

【態度3: drudgery vs 思考活動を区別】
  ├─ 修正 → テスト → 失敗 → 修正...
  │   └─ 思考活動、必要な投資 → 続ける
  └─ 進む

【態度4: テスト結果（現実）を信じる】
  ├─ テストエラー出現
  │   └─ 理論を疑う → 原因調査
  └─ テスト成功 → 進む

【態度5: 能力を検証する】
  ├─ テスト結果 = エラー
  │   └─ 能力不足と認識 → 学習へ
  └─ テスト結果 = 成功 → 完了
```
