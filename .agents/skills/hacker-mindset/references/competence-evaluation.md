# 能力の評価方法

**出典**: ESR 「Attitude is no substitute for competence」  
**目的**: 自分の能力を正確に把握し、改善すべき領域を特定する

---

## 概要

「正しい態度」を持つことと「実装する能力」は別である。能力不足を認識してこそ、初めて改善できる。

---

## 能力評価の 5 段階

### Level 1: 未熟（Novice）

**特徴**：
- エラーが出ても、なぜ出ているか理解できない
- ドキュメントを読んでも理解できない
- 修正方法が分からない
- テストの書き方が分からない

**判断基準**：
```
エラー出現
  ↓
【認識】「これは何か分からない」
  ↓
修正できない、ドキュメントを読んでも分からない
```

**改善アクション**：
- 基礎的なチュートリアル実施
- 他者のコードを読む
- 同じエラーの解決例を探す
- 経験者に質問

---

### Level 2: 初心者（Beginner）

**特徴**：
- エラーが出た時、ドキュメントを読むことで分かるようになった
- 過去に同じエラーを経験したら解決できる
- しかし、根本原因を理解していない
- テストは書けるが、不十分

**判断基準**：
```
エラー1回目
  ↓
【対応】ドキュメント読む、Google 検索 → 解決

エラー2回目（同じエラー）
  ↓
【対応】過去の解決方法を思い出す → 解決

エラー3回目（同じエラーだが文脈が異なる）
  ↓
【判断】「これは同じ原因か別の原因か？」判定できない
```

**改善アクション**：
- 同じエラーを 3 回経験したら、根本原因ドキュメント作成
- エラーの背景にある仕様を学ぶ
- テストを増やす
- コードレビューで指摘を受ける

---

### Level 3: 習熟（Intermediate）

**特徴**：
- エラーが出たら、すぐに根本原因を判定できる
- 類似エラーは1回で解決できる
- テストを設計できる
- 予防的にコード品質を高められる

**判断基準**：
```
エラー出現
  ↓
【判断】「この原因はX、Y、Z のどれか」と複数候補を想定
  ↓
テストで原因特定
  ↓
修正 + テスト

新しいエラーは 1 回で原因特定できる
```

**改善アクション**：
- エラーが出ないように設計段階で配慮
- 他の初心者に教える
- ドキュメント、チェックリスト作成
- アーキテクチャレベルの最適化

---

### Level 4: 熟練（Advanced）

**特徴**：
- エラーが出る前に問題を予測できる
- 複数言語/フレームワークで同じ能力を発揮
- 他者のコードの潜在的問題を指摘できる
- チーム全体の能力を高められる

**判断基準**：
```
コード実装
  ↓
【予測】「この設計だと、こんなエラーが出そう」
  ↓
【対応】事前に対策を施す
  ↓
実装完了 → エラーが出ない
```

---

### Level 5: 達人（Expert）

**特徴**：
- 新しい技術でもすぐに習熟できる
- 深い知識で複雑な問題を解決
- 学習方法を最適化できる
- 業界標準を定められる

---

## 自己評価チェックリスト

### 基本スキル評価

```
□ 設定ファイル（YAML, JSON, Terraform, CloudFormation）
  ├─ 構文エラーが出たら、原因を特定できるか？
  ├─ 複数の類似エラーパターンを区別できるか？
  └─ 予防的にチェックできるか？
  
□ プログラミング言語（Python, Go, TypeScript）
  ├─ ビルドエラーが出たら、原因を特定できるか？
  ├─ ロジックエラーを読み取れるか？
  └─ テストケースを設計できるか？

□ API（REST, GraphQL）
  ├─ ステータスコードエラーの原因を特定できるか？
  ├─ リクエスト形式の誤りを検出できるか？
  └─ 仕様との不一致を判定できるか？

□ インフラ（EC2, RDS, Lambda, Networking）
  ├─ デプロイ失敗の原因を特定できるか？
  ├─ ログから問題を追跡できるか？
  └─ 事前に設定を検証できるか？

□ データベース（SQL, DynamoDB）
  ├─ クエリエラーの原因を特定できるか？
  ├─ パフォーマンス問題を診断できるか？
  └─ スキーム設計の誤りを指摘できるか？
```

## 能力評価フレームワーク

### 各領域での能力レベル判定

```
【作業を開始する前】

Q1: この作業に必要な知識はあるか？
  ├─ YES → 確信を持って実装可能
  ├─ PARTIAL → 調べながらできそう
  └─ NO → 学習が必要

Q2: 同じ種類の作業を過去にしたことがあるか？
  ├─ 複数回 → Level 3 以上
  ├─ 1 回 → Level 2
  └─ なし → Level 1

【作業中】

Q3: エラーが出たか？
  ├─ YES → Q4 へ
  └─ NO → 続行

Q4: 原因を特定できたか？
  ├─ YES（即座に） → Level 3 以上
  ├─ YES（時間がかかった） → Level 2
  └─ NO → Level 1

【作業完了後】

Q5: 同じエラーは二度と出ないか？
  ├─ テスト追加した、ドキュメント作成した → Level 3 以上
  ├─ 同じ修正方法を記録した → Level 2
  └─ まだ理解していない → Level 1

Q6: 他の人にこの知識を教えられるか？
  ├─ YES → Level 4 以上
  ├─ PARTIAL → Level 3
  └─ NO → Level 1-2
```

## 段階別の改善計画

### Level 1 → 2: 基礎習得

```
目標: エラーの解決方法を知る

実行項目:
  □ 公式ドキュメント読破
  □ チュートリアル完了
  □ 5 つ以上の異なるエラーを経験
  □ 各エラーの解決方法を記録
  □ 同じエラーを 1 回で解決できるようになる

期間: 3-6 ヶ月
```

### Level 2 → 3: 根本原因理解

```
目標: エラーの根本原因を理解する

実行項目:
  □ 同じエラーが 3 回出たら根本原因ドキュメント作成
  □ エラーの「背景にある仕様」を学ぶ
  □ テストケースを 3 倍増やす
  □ コードレビュー（受ける側）で指摘を記録
  □ 予防的なチェックリスト作成

期間: 6-12 ヶ月
```

### Level 3 → 4: 予防的能力

```
目標: エラーが出る前に問題を予測する

実行項目:
  □ コードレビュー（指摘する側）で練習
  □ アーキテクチャレベルの設計検討
  □ 複数言語で同じ能力を発揮
  □ チーム用ドキュメント/チェックリスト作成
  □ 初心者に教える

期間: 1-2 年
```

## エラー対応から見た能力評価

```
状況                                    能力レベル
─────────────────────────────────────────────────
エラー出現 → 原因分からない              Level 1
エラー出現 → ドキュメント見て解決         Level 2
エラー出現 → すぐに原因特定 → 修正        Level 3
エラー出現を予測 → 事前に対策            Level 4
複数言語/領域で Level 4 を発揮          Level 5
```

---

## 「能力がない」ことを認識する判断基準

```
□ 同じセクションで 2 回以上修正失敗
  → 根本原因がまだ理解されていない
  → 能力不足を認識すべき

□ エラーメッセージを読んでも分からない
  → 用語、概念の知識不足
  → 学習投資が必要

□ 過去に同じエラーを経験したはずなのに忘れた
  → 知識定着が不十分
  → ドキュメント化すべき

□ 他者の指摘で「あ、そっか」と気づく
  → 知識はあるが応用できていない
  → 経験が不足している

□ 3 回以上同じミスをしている
  → テストまたはチェックリストが不足
  → プロセス改善が必要
```

## 能力向上の具体的ステップ

### ステップ 1: 現在地を把握

```
各領域を 1-5 で自己評価

CloudFormation  : Level ___
Python          : Level ___
API 設計        : Level ___
インフラ設計    : Level ___
データベース    : Level ___
```

### ステップ 2: 改善優先順位を決定

```
次に改善すべき領域は？
  ├─ Level 1 の領域を Level 2 に
  ├─ Level 2 の領域を Level 3 に
  └─ Level 3 の領域を Level 4 に
```

### ステップ 3: 改善実行

```
【改善方法】
  ├─ エラーが出たら根本原因ドキュメント作成
  ├─ チェックリスト追加
  ├─ テストケース追加
  ├─ ドキュメント更新
  └─ 他者に教える
```

### ステップ 4: 定期的に再評価

```
3 ヶ月ごとに自己評価を更新
  ├─ Level が上がったか？
  ├─ 同じエラーは減ったか？
  └─ 次の改善領域を決定
```

---

## 重要な判断基準

```
✅ 能力がある = テスト結果で証明される
❌ 能力がない = エラーが出ている

「態度は正しい」と感じても、
テスト結果がエラーを示していたら、
それは能力不足である。
```
