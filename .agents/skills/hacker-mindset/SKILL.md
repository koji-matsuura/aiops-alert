# Hacker Mindset Skill

**著者**: Eric S. Raymond（How To Become A Hacker）  
**実装**: 実践的なワークフロー化  
**目的**: 理論ではなく、実際の問題解決で使える判断基準を提供

---

## 概要

このスキルは、ESR の古典「How To Become A Hacker」を実践的な判断ツールに変換しています。目的は：

- ✅ **理論的正確性より実践的判断を優先**
- ✅ **エラーや問題に直面した時の対応を明確化**
- ✅ **試行錯誤と根本原因調査を奨励**
- ✅ **自分の修正・判断を常に検証する**
- ✅ **「完遂したい」という欲望に流されない**

---

## コア原則（5 つのハッカー態度）

### 1. The world is full of fascinating problems waiting to be solved

**実装レベル**：エラーや問題が出現したら、それは「乗り越えるべき障害」ではなく「解くべき問題」である。

**判断基準**：
```
エラー出現
  ↓
【判定】「これは何か？」「なぜ起きているか？」と問う
  ↓
【アクション】原因を徹底的に調査する
  ↓
【結果】問題の本質を理解する
```

**反パターン**（私が犯した誤り）：
```
エラー出現
  ↓
【判定】「これは解決する必要のない問題」と判定
  ↓
【アクション】別の場所を修正して進める
  ↓
【結果】問題が悪化する
```

---

### 2. No problem should ever have to be solved twice

**実装レベル**：同じエラーで複数回失敗することは許されない。1 回目のエラーで原因を完全に把握する。

**判断基準**：
```
エラー1回目出現
  ↓
【投資】十分な時間をかけて根本原因を調査
  ↓
【記録】原因と解決策を記録
  ↓
【適用】同じエラーは二度と発生させない
```

**反パターン**（私が犯した誤り）：
```
cfn-lint エラー1回目
  ↓
修正してみる（原因未調査）
  ↓
cfn-lint エラー2回目
  ↓
別の修正をしてみる（根本原因未把握）
  ↓
cfn-lint エラー3回目以上 ← この時点で異常に気づくべき
```

---

### 3. Boredom and drudgery are evil

**実装レベル**：「これは何度も繰り返される作業か？」と問う。そうなら自動化を考える。

ただし：**試行錯誤は drudgery ではない。必要な思考活動である。**

**判断基準**：
```
修正作業
  ├─ 試行錯誤（思考・学習） → ✅ 必要
  └─ 同じ手順の繰り返し → ❌ 自動化対象
```

---

### 4. Freedom is good

**実装レベル**：**権威（教師評価、理論的正確性）に盲従しない。**

自分の実装が「理論的に正しい」と感じても、実際のエラーが出現したら、理論を捨てて現実を優先する。

**判断基準**：
```
理論的には正しいはずの修正
  ↓
テスト実行（cfn-lint など）
  ↓
【結果がエラー】
  ↓
【即座に判定】理論が間違っているか、実装が間違っているか
  ↓
【権威を無視】修正を止めて原因調査
```

**反パターン**（私が犯した誤り）：
```
「IAM ロール命名統一は正しい設計」という理論
  ↓
cfn-lint エラー出現
  ↓
【判定】「理論は正しい、別の箇所が問題」
  ↓
修正を重ねる ← 理論への盲従
```

---

### 5. Attitude is no substitute for competence

**実装レベル**：「正しい態度」を持つことと「正しく実装する能力」は別である。

**判断基準**：
```
自分の修正が「正しい」と感じる
  ↓
【必ず検証する】テストツール（cfn-lint、pytest など）で確認
  ↓
【エラーがない】OK
【エラーがある】能力不足 → 原因調査へ
```

**重要**：エラーが出た = 「態度不足」ではなく「能力不足」である。態度を変えるのではなく、能力を高める。

## 実践ワークフロー

### Phase 1: 問題検出

```
エラー・問題が出現した
  ↓
【判定】このエラーの重大性は？
  ├─ デプロイ不可レベル（cfn-lint error）
  │   → 【即座に対応】修正を止めて原因調査
  │
  ├─ 警告レベル（lint warning）
  │   → 【判断】この警告は重要か？
  │       ├─ YES → 修正
  │       └─ NO → 無視して続行
  │
  └─ 情報レベル（lint info）
      → 【判定】後で対応してもいい
          → ログに記録して続行可
```

### Phase 2: 原因調査（カギになるステップ）

**このステップを飛ばしてはいけない。**

```
エラー出現
  ↓
【質問1】「このエラーは私の修正が原因か？」
  ├─ YES → Phase 3 へ
  └─ UNCLEAR → Phase 2a へ
  
Phase 2a: 修正と エラー出現の因果関係を確認
  ├─ git diff で修正内容を確認
  ├─ 修正前後の差分を調査
  ├─ 修正が当該エラーに影響するか判定
  └─ 【結論】YES / NO を確定

  ├─ YES（修正が原因）
  │   ↓
  │   【質問2】「修正の何が問題か？」
  │   ├─ 構文エラー？
  │   ├─ ロジックエラー？
  │   ├─ 前提条件の誤り？
  │   ├─ 仕様理解の誤り？
  │   └─ 【根本原因を特定】
  │       ↓
  │       Phase 3 へ
  │
  └─ NO（修正が原因ではない）
      ↓
      【判定】では、何が原因か？
      ├─ 別の人の修正？
      ├─ 既存の問題？
      ├─ 環境の問題？
      └─ 別途調査
```

### Phase 3: 根本原因の詳細分析

```
根本原因が特定された
  ↓
【分析】なぜその原因が生じたのか？
  ├─ 知識不足？
  │   └─ 公式ドキュメントを読む
  │
  ├─ 実装ミス？
  │   └─ コードをもう一度見直す
  │
  ├─ 設計ミス？
  │   └─ アーキテクチャを再検討
  │
  ├─ 仕様理解の誤り？
  │   └─ AWS ドキュメントを確認
  │
  └─ 前提条件の誤り？
      └─ 環境設定を確認
```

### Phase 4: 修正

```
根本原因が理解された
  ↓
【修正】原因に対する修正を実装
  ├─ 最小限の修正か？（余計な変更なし）
  └─ 修正が根本原因を解決するか？

修正実装
  ↓
【即座に検証】テストツールで確認
  ├─ cfn-lint エラー消滅？
  ├─ 新しいエラー発生？
  ├─ 既存機能に影響？
  └─ 【判定】全て OK か？
```

### Phase 5: 学習記録

```
問題が解決した
  ↓
【記録】同じエラーは二度と繰り返さない
  ├─ 原因は何か？
  ├─ 解決策は何か？
  ├─ なぜこの誤りが起きたのか？
  └─ 次回の予防策は？

【適用】同じパターンのコードを確認
  ├─ 同じ誤りが他の箇所にないか？
  └─ 他の箇所も修正
```

## 意思決定ツリー

### エラーに直面した時

```
エラー出現
  ↓
【質問】このエラーを修正すべきか？
  ├─ デプロイ失敗エラー（cfn-lint error）
  │   └─ YES, 必ず修正 → Phase 1 開始
  │
  ├─ 警告（cfn-lint warning）
  │   ├─ セキュリティ関連？
  │   │   └─ YES, 修正 → Phase 1 開始
  │   └─ スタイル関連？
  │       ├─ 重要？ → 修正
  │       └─ 軽微？ → ログ記録
  │
  └─ 情報（cfn-lint info）
      └─ 後回し、ログ記録
```

### 修正が複数回失敗した時

```
同じセクションで 2 回以上修正失敗
  ↓
【警告】何かが根本的に間違っている
  ├─ 【対応】git diff でこれまでの修正を全て確認
  ├─ 【判定】修正の履歴から何が起きているか分析
  ├─ 【決定】修正を一度全部戻すべきか？
  │   ├─ YES → git checkout で元に戻す
  │   └─ NO → 代替アプローチを検討
  └─ 【再開始】Phase 1 から開始
```

### 「完遂したい」という欲望に流された時

```
「この修正を完成させたい」という感情
  ↓
【警告】この欲望は敵である
  ├─ cfn-lint エラーが出ている
  ├─ 複数回の修正で解決していない
  ├─ ユーザーが正しく修正した
  └─ 【判定】「完遂」ではなく「原因理解」を優先
      ↓
      修正を止める
      ↓
      原因調査に切り替える
```

## 反パターン（避けるべき判断）

### 反パターン 1: 理論への盲従

```
❌ 誤り：
「CloudFormation はコード = Linter エラーは修正すべき」（理論）
  ↓
cfn-lint エラー出現
  ↓
【判定】理論が正しい、別に問題がある
  ↓
修正を重ねる

✅ 正しい：
「cfn-lint エラー出現 = 修正が失敗した」（現実）
  ↓
理論を検証する前に、現実を優先
  ↓
原因を調査する
```

### 反パターン 2: 試行錯誤を drudgery と勘違い

```
❌ 誤り：
「修正 → テスト → 失敗 → 修正...」は drudgery だから避ける
  ↓
修正を1回だけして検証を飛ばす

✅ 正しい：
試行錯誤は「思考活動」であり、必要な投資
  ↓
何度でも修正して、最後に「完全に理解された状態」に到達
```

### 反パターン 3: ユーザーの判断を無視

```
❌ 誤り：
ユーザーが正しく修正した
  ↓
【判定】「まだ不完全なのでは？」と疑う
  ↓
さらに修正しようとする

✅ 正しい：
ユーザーが修正した
  ↓
その修正が正しいか確認する（検証）
  ↓
正しければ、そこで止める
```

## 指標：あなたはハッカーか？

ESR の判定基準：

```
【質問1】スキルはあるか？
  ├─ YES → 次へ
  └─ NO → スキルを磨く

【質問2】ハッカー態度を理解しているか？
  ├─ YES → 次へ
  └─ NO → このスキルを学ぶ

【質問3】他のハッカーが「お前はハッカーだ」と認めるか？
  ├─ YES → あなたはハッカー
  └─ NO → ステータス獲得フェーズへ
```
